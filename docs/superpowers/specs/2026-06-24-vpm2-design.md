# VPM2 — Vídeos Para Minha Mãe (design)

**Data:** 2026-06-24
**Status:** Aprovado para planejamento

## Objetivo

Traduzir vídeos do YouTube de inglês para português (PT-BR) para uso pessoal —
tornar vídeos (ex: tutoriais de pintura do Bob Ross) assistíveis para a mãe do
autor. Todo o processamento de ASR, tradução, TTS e edição de áudio é feito
**localmente**.

Caso de uso de referência: traduzir um vídeo de pintura do Bob Ross.

## Escopo da v1

- **Interface:** CLI (`vpm2 <url>`). UI web é uma fase futura, construída como
  camada fina sobre o mesmo pipeline.
- **ASR:** um modelo forte (sem ensemble). Ensemble de ASR + LLM juiz é fase futura.
- **Locutor:** assume **um único locutor**. Diarização multi-locutor é fase futura.
- **Voz (TTS):** backend plugável com dois modos — voz PT pronta (*preset*) e voz
  **clonada** do locutor original — para comparação.
- **Sincronismo:** por trecho, com ajuste leve (consumir pausas → acelerar clipe
  com teto → empurrar próximo segmento). Sincronismo agressivo é fase futura.

## Ambiente de execução

- **Sistema:** WSL2 (Ubuntu) com passthrough de GPU, dentro do Windows do autor.
  O projeto vive em `C:\Users\PVS\projetos\vpm2`, acessível do WSL em
  `/mnt/c/Users/PVS/projetos/vpm2`.
- **Hardware:** RTX 5060 Ti 16GB VRAM (Blackwell / sm_120). Requer PyTorch + CUDA
  recentes (cu128/nightly) — pegadinha de setup a prever na instalação.
- **Linguagem:** Python. Gerenciador de ambiente: `uv` ou `conda` (definir no setup).

## Ferramentas externas

- `yt-dlp` — download do vídeo.
- `ffmpeg` — extração de áudio, ajuste de tempo (`atempo`), mux final.
- **Ollama** — roda o LLM de tradução local (caminho de menor atrito no ecossistema;
  também prepara o terreno para o "LLM juiz" da fase futura).

## Arquitetura

Pipeline em **estágios com artefatos em disco**. Cada estágio lê uma entrada,
escreve um artefato em `work/<video-id>/`, e o próximo estágio lê esse artefato.
Permite **retomada automática**, **inspeção** de cada etapa e **troca de modelo**
sem reescrever o pipeline.

```
01_video.mp4 → 02_audio.wav → 03_transcript.json → 04_translation.json
   → 05_clips/*.wav → 06_final.mp4
```

### Estrutura do projeto

```
vpm2/
├─ pyproject.toml          # deps + entrypoint da CLI
├─ vpm2/
│  ├─ cli.py               # `vpm2 <url>` — orquestra os estágios
│  ├─ pipeline.py          # roda estágios em ordem, gerencia retomada
│  ├─ config.py            # modelos, idioma, voz, params de sync
│  ├─ stages/
│  │  ├─ base.py           # interface Stage (contrato comum)
│  │  ├─ download.py       # 01 — yt-dlp
│  │  ├─ extract_audio.py  # 02 — ffmpeg
│  │  ├─ transcribe.py     # 03 — ASR (EN→texto EN c/ timestamps)
│  │  ├─ translate.py      # 04 — LLM (EN→PT)
│  │  ├─ synthesize.py     # 05 — TTS (PT→clipes de áudio)
│  │  └─ assemble.py       # 06 — sync + mux final (ffmpeg)
│  └─ tts/
│     ├─ base.py           # interface TTSBackend
│     ├─ cloning.py        # voz clonada
│     └─ preset.py         # voz PT pronta
├─ work/<video-id>/        # artefatos intermediários (gitignored)
└─ docs/superpowers/specs/ # design docs
```

### Contrato de um estágio

```python
class Stage:
    name: str
    def output_path(self, ctx) -> Path: ...   # ex: work/<id>/03_transcript.json
    def is_done(self, ctx) -> bool: ...        # artefato existe E é válido?
    def run(self, ctx) -> None: ...            # produz o artefato
```

`pipeline.py` percorre os estágios em ordem: se `is_done()` é verdadeiro, **pula**
(retomada); senão `run()`. Flag `--force <stage>` reprocessa a partir de um ponto.

`TTSBackend` é uma interface separada (`cloning` vs `preset`), selecionada por
config/flag — permite comparar as duas vozes sem mexer no pipeline.

## Estágios em detalhe

### 01 · Download
`yt-dlp` baixa o melhor mp4. Salva `01_video.mp4` + metadados (título, duração,
id). O id nomeia a pasta de trabalho `work/<video-id>/`.

### 02 · Extrair áudio
`ffmpeg` extrai `02_audio.wav` em 16kHz mono (formato esperado pelo ASR).

### 03 · Transcrição (ASR)
**`faster-whisper` large-v3** (backend CTranslate2 — rápido na GPU, baixo VRAM).
Produz `03_transcript.json`: lista de segmentos `{start, end, text}` em inglês.
Os timestamps por segmento são a base do sincronismo.
*Upgrade futuro plugável: WhisperX para alinhamento por palavra.*

### 04 · Tradução (LLM via Ollama)
Traduz os segmentos EN → PT-BR **preservando a segmentação** (frase a frase, com
contexto das frases vizinhas para coerência). Modelos candidatos: **Qwen2.5-7B-Instruct**
ou **Gemma2-9B** (bom PT, cabem folgado em 16GB). Saída `04_translation.json`:
mesmos segmentos com `text_pt`. Prompt instruído para tradução natural/falada
(não literal), estilo narração.

### 05 · Síntese (TTS)
Para cada segmento, gera um clipe PT via o `TTSBackend` selecionado. Salva
`05_clips/0001.wav`, `0002.wav`... + `05_clips.json` com a duração real de cada
clipe. Modelos candidatos (ambos fazem PT e suportam preset + clonagem):
- **XTTS-v2** (Coqui) — multilíngue, vozes preset e clonagem por amostra.
- **Chatterbox Multilingual** — PT + clonagem zero-shot.

Começar com um deles por trás de `TTSBackend`, expondo os dois modos. Escolha
definida no setup, testando com a voz da narração.

**Amostra para clonagem:** extraída **automaticamente** do próprio vídeo — um
trecho contínuo (~10s) só com voz, escolhido via detecção de fala (VAD) para
evitar música/silêncio.

### 06 · Montagem e sincronismo
Para cada segmento temos `start`/`end` original (ASR) e a duração real do clipe PT
(TTS). Algoritmo:

1. Construir uma linha do tempo vazia com a duração do vídeo original.
2. Posicionar cada clipe PT no `start` do seu segmento.
3. Quando o clipe PT estoura o tempo disponível até o próximo segmento:
   - consumir pausas/silêncio entre segmentos (folga grátis);
   - se ainda estoura, **acelerar o clipe** com `ffmpeg atempo` com **teto**
     (ex: 1.2–1.25x) para não distorcer a voz;
   - se mesmo no teto estoura, **empurrar o próximo segmento** um pouco (o atraso
     tende a se reabsorver nas próximas pausas).
4. Concatenar os clipes posicionados em `06_audio_pt.wav`.
5. **Mux final** (`ffmpeg`): copia o vídeo original, **substitui a trilha de áudio**
   pela PT → `06_final.mp4`. Mantém o **áudio original como segunda trilha mutada**
   (custo ~zero; ajuda comparação/depuração).

**Parâmetros configuráveis** (`config.py`): teto de aceleração, permitir empurrar
segmentos, margem de pausa. Gancho pronto para o sincronismo agressivo futuro.

## Tratamento de erros e robustez

- **Retomada por artefato:** falha em um estágio não perde o trabalho anterior;
  reexecutar continua de onde parou. `--force <stage>` reprocessa de um ponto.
- **Validação de artefato:** `is_done()` valida minimamente (JSON parseável,
  nº de clipes == nº de segmentos). Artefato parcial/corrompido é descartado e refeito.
- **Falhas externas:** yt-dlp (vídeo indisponível/idade), Ollama offline, VRAM
  insuficiente — cada estágio dá erro claro e acionável, não stack trace cru.
- **Modelos sob demanda:** carregados e liberados entre estágios pesados para não
  competir por VRAM (ASR → LLM → TTS, sequencial; cabe folgado em 16GB).
- **Logs por etapa** em `work/<id>/logs/`.

## Testes

- **Unitários** na lógica pura (maior risco): algoritmo de sync (posicionamento /
  aceleração / empurrão a partir de segmentos+durações fake), parsing/validação de
  artefatos, montagem do prompt de tradução. Sem GPU, rápidos.
- **Contrato dos backends:** testes garantindo que `TTSBackend` (preset e cloning)
  respeitam a mesma interface.
- **Smoke test ponta-a-ponta** com clipe curto (~30s) e modelos reais —
  manual/opcional (depende de GPU e download), documentado no README.
- Estágios que são casca fina sobre ferramentas externas (yt-dlp/ffmpeg) não são
  alvo de teste pesado — testamos a nossa lógica, não a ferramenta.
